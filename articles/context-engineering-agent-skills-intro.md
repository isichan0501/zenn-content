## TL;DR
「Agent-Skills-for-Context-Engineering」は、プロンプト単体の工夫ではなく **“コンテキスト全体の設計”** でAIエージェントの性能を底上げするためのスキル集です。
注意（Attention）の物理的制約（例：*lost-in-the-middle*）を前提に、段階的開示・メモリ・評価などの設計パターンをまとめています。

## この記事で扱うこと
- コンテキスト・エンジニアリングとは何か（プロンプトエンジニアリングとの違い）
- リポジトリから読み取れる主要コンセプト（10個）
- まず試せる「最小構成」（段階的開示・評価・メモリ）

## なぜ今「コンテキスト・エンジニアリング」なのか
LLMのコンテキスト窓が大きくなっても、モデルがすべてを均等に見てくれるわけではありません。
長い入力では重要情報が埋もれたり、途中が弱くなったりします（いわゆる *lost-in-the-middle*）。

だから「良い指示を書く」だけでなく、
- どの情報を
- どの順番で
- どれくらいの粒度で
- いつ読み込ませるか
を **設計** する必要が出てきます。

## リポジトリ概要
ソース: https://github.com/muratcankoylan/Agent-Skills-for-Context-Engineering

このリポジトリは、AIエージェントを“実運用レベル”に持っていくための「コンテキスト設計の型」を集めたもの。
Claude CodeやCursorなどの環境で使うことを想定しつつ、プラットフォームに依存しない原則として整理されています。

## 主要コンセプト（ざっくり10個）
- コンテキスト・エンジニアリング
- 注意のメカニズムと減衰（lost-in-the-middle）
- プログレッシブ・ディスクロージャ（段階的開示）
- マルチエージェント・アーキテクチャ
- メモリ・システム設計
- プラットフォーム非依存
- LLM-as-a-Judge（評価者としてのLLM）
- BDIメンタルステート（信念/欲求/意図）
- スキル・トリガー
- プロダクション・グレードの評価

:::message
「強いプロンプト」を探すより、**“強いコンテキスト供給パイプライン”** を作った方が再現性が高い。
:::

## まず試す：段階的開示（Progressive Disclosure）の最小構成
「全部入れる」だと、
- コストが上がる
- 重要情報が埋もれる
- デバッグが難しくなる
ので、次の2層に分けます。

- **Layer 1（常時）**：短いルール + 超要約
- **Layer 2（必要時）**：詳細（仕様、過去ログ、設計資料、長文）

### 例：フォルダで分ける
（実装は何でもいいけど、“分け方”のイメージが大事）

```txt
context/
  always.md        # 常に渡す（短い）
  on_demand/
    api.md         # 必要になったら渡す
    schema.md
    troubleshooting.md
```

### 例：プロンプト側の疑似コード

```txt
if task in {"debug", "design"}:
  include(always.md)
  include(on_demand/troubleshooting.md)
else:
  include(always.md)
```

## 評価を先に作る（LLM-as-a-Judge）の最小構成
エージェントを強くする一番簡単な方法は「評価を作って回す」ことです。

### 例：評価ルーブリック（超ミニ）
- 正確性（根拠があるか）
- 再現性（別入力でも同程度の品質か）
- 失敗の仕方（嘘をつかないか）
- コスト（トークン/時間）

:::details なぜ“実装前”に評価を作るのか
あとから評価を作ると、作った実装に評価が引っ張られて「都合の良い採点」になりがち。
先に評価軸を固定すると、改善ループが回ります。
:::

## メモリの置き場所を決める（最小構成）
会話履歴に全部頼るのは不安定になりがち。
最低限、次のどれかに逃がすだけでも差が出ます。

- “決定事項”だけを `decisions.md` に追記
- よく使う手順だけを `runbook.md` に追記
- 参考リンクを `sources.md` に追記

## チェックリスト（今日から使う）
- [ ] コンテキストを「常時」と「必要時」に分けた
- [ ] 重要情報を先頭に寄せた（中盤に埋めない）
- [ ] 評価軸（ルーブリック）を先に作った
- [ ] メモリの置き場所（ファイル/DB）を1つ決めた

## まとめ
このリポジトリは「コンテキストを雑に増やす」のではなく、注意資源を前提に **情報の流れを設計する** 方向へ、エージェント開発を引っ張る資料です。

次は、特定のトピック（例：段階的開示、評価、メモリ）に絞って、自分のワークスペースに“最小構成”を入れてみると理解が早いです。
